# Architecture

## System Overview

```
┌──────────────┐     ┌──────────────────┐     ┌──────────────┐
│   Frontend   │────▶│   Go REST API    │────▶│  PostgreSQL  │
│  Vite/React  │◀────│   (Chi Router)   │◀────│     16       │
│  :5173       │     │   :8000          │     │   :5432      │
└──────────────┘     └──────────────────┘     └──────────────┘
```

The platform follows a classic client-server architecture. The React SPA communicates with the Go API over HTTP/JSON. The API owns all business logic and data access.

## Backend Architecture

The backend follows a layered architecture with clear separation of concerns:

```
HTTP Request
    │
    ▼
┌─────────────────┐
│   Router        │  Route definitions and middleware chain
│   (Chi)         │
└────────┬────────┘
         │
    ▼
┌─────────────────┐
│   Middleware     │  Auth, CORS, logging, rate limiting, recovery
└────────┬────────┘
         │
    ▼
┌─────────────────┐
│   Handler       │  HTTP concerns: parse request, validate, call service, write response
└────────┬────────┘
         │
    ▼
┌─────────────────┐
│   Service       │  Business logic, orchestration, domain rules
└────────┬────────┘
         │
    ▼
┌─────────────────┐
│   Repository    │  Data access, SQL queries, database interaction
└────────┬────────┘
         │
    ▼
┌─────────────────┐
│   PostgreSQL    │  Data persistence
└─────────────────┘
```

### Layer Responsibilities

**Router** — Defines routes, groups them by domain, applies middleware. No business logic here.

**Middleware** — Cross-cutting concerns applied to the request pipeline. Each middleware does one thing. Auth middleware extracts and validates JWT tokens. Logging middleware records request duration and status.

**Handler** — Translates between HTTP and the domain. Parses request bodies, validates input, calls the appropriate service method, and formats the response. Handlers should not contain business logic.

**Service** — Contains all business rules and orchestration. Coordinates between multiple repositories if needed. This is where domain logic lives. Services are unit-testable with mocked repositories.

**Repository** — Encapsulates all database interactions. Uses sqlc-generated code for type-safe queries. One repository per database table or aggregate.

### Dependency Injection

Dependencies flow downward and are injected via struct fields:

```go
type UserHandler struct {
    service *UserService
}

type UserService struct {
    repo   *UserRepository
    mailer *EmailService
}

type UserRepository struct {
    db *pgxpool.Pool
}
```

All dependencies are wired together in `cmd/server/main.go` at startup.

## Frontend Architecture

The frontend follows a feature-based organization:

```
src/
├── api/           # Axios instance, endpoint functions, interceptors
├── components/    # Shared, reusable UI components (Button, Modal, etc.)
├── features/      # Feature modules (auth, dashboard, billing, etc.)
│   └── auth/
│       ├── components/    # Feature-specific components
│       ├── hooks/         # Feature-specific hooks
│       ├── pages/         # Route-level pages
│       └── types.ts       # Feature-specific types
├── hooks/         # Shared custom hooks
├── layouts/       # Page layout wrappers
├── pages/         # Top-level route pages
├── stores/        # Global state (Zustand)
├── types/         # Shared TypeScript types
└── utils/         # Helper functions
```

### Data Flow

```
User Action → Component → API Client → Backend → Response → State Update → Re-render
```

The API client in `src/api/` centralizes all HTTP communication. Axios interceptors handle token attachment and automatic refresh on 401 responses.

## Database Design Principles

- Every table has a UUID `id` primary key generated by the database
- `created_at` and `updated_at` timestamps on all tables, managed by triggers
- Soft deletes via `deleted_at` column where needed
- Foreign keys with explicit `ON DELETE` behavior (CASCADE, SET NULL, or RESTRICT)
- Indexes on columns used in WHERE clauses and JOINs
- Use database-level constraints (NOT NULL, UNIQUE, CHECK) as the last line of validation

## Authentication and Authorization

### Authentication Flow

```
1. POST /api/v1/auth/register  →  Create account, return tokens
2. POST /api/v1/auth/login     →  Verify credentials, return tokens
3. POST /api/v1/auth/refresh   →  Exchange refresh token for new access token
4. POST /api/v1/auth/logout    →  Invalidate refresh token
```

Access tokens are short-lived JWTs (15–60 min). Refresh tokens are long-lived and stored in the database for revocation capability.

### Authorization

Role-based access control (RBAC) is enforced at the middleware and service layers. Roles are stored in the database and included in the JWT claims.

## API Design

- RESTful endpoints under `/api/v1/`
- JSON request and response bodies
- Consistent response envelope: `{ "data": ..., "error": ..., "meta": ... }`
- Pagination via `?page=1&per_page=20` with total count in `meta`
- Filtering via query parameters: `?status=active&sort=created_at&order=desc`
- HTTP status codes used correctly (201 for creation, 204 for deletion, etc.)

## Error Handling Strategy

### Backend

Custom error types in `internal/model/errors.go` map to HTTP status codes. The error middleware catches these and returns a consistent JSON response. Unexpected errors return 500 with a generic message (details logged server-side only).

### Frontend

Axios interceptors catch errors globally. Network errors and 5xx show a toast notification. 401s trigger token refresh. 4xx errors are handled at the component level for field-specific feedback.

## Deployment

### Local Development

Docker Compose orchestrates PostgreSQL and any other services. The Go server and Vite dev server run natively for hot-reload.

### CI/CD

GitHub Actions runs on every PR: linting, testing, and building for both backend and frontend. Merges to `main` trigger deployment.
